# Bundled fonts (Batch 3.2 / certificate PDF generation)

Two static TrueType fonts at the SemiBold weight, bundled into the
api-server build for use by [`certificate-pdf.ts`](../../services/certificate-pdf.ts).

The certificate is generated by overlaying 4 dynamic text fields onto
[`../certificate-template.pdf`](../certificate-template.pdf) — Faris's
designed template carries all the visual identity (logo, decorations,
signature, etc.). Only SemiBold is needed for the overlays:

- **Lora-SemiBold** — holder name (28pt)
- **Inter-SemiBold** — score and date values (18pt + 14pt)

The UUID overlay uses pdf-lib's built-in `StandardFonts.Courier` (every
PDF reader has it built in; no asset file needed).

## Why static TTFs instead of variable fonts

The Google Fonts main repo only ships variable-axis `.ttf` for both Lora and Inter (single file per font, weight selected via the `wght` axis). pdf-lib + fontkit can register the variable font at the default weight, but the named-instance lookup via `embedFont(bytes, { instance: "SemiBold" })` triggers a subsetting bug at PDF finalization (`TypeError: First argument to DataView constructor must be an ArrayBuffer`).

Static TTFs (one file per weight) sidestep the variable-font code path entirely. The bundle cost is ~638 KB instead of ~1.1 MB for the variable fonts — net win.

## v1 limitations (English-only)

These fonts cover Latin + a handful of extended-Latin code blocks. Non-Latin scripts (Arabic, Hebrew, CJK, Devanagari) render as missing-glyph boxes. This is the v1 scope decision (Batch 3.2 DP3). Post-launch i18n batch should add NotoSans CJK + NotoSans Arabic + NotoSans Devanagari + NotoSans Hebrew for the full 14-language UI matrix.

Smoke test #5 (CJK holder name "张伟") verified the renderer degrades gracefully (boxes, not crash) under the missing-glyph case.
